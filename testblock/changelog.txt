    CLAUDE: more on triple multiply
            Stage 2b (Linear Projections) and Stage 4 (Down Projection) — 
            specifically inside the Q4K MAC inner loop, line 186:
            int_acc_w[b][k] += (int32_t)(xi8 * (ap_int<5>)nib4 * (ap_int<7>)sc6u);
            This is the Q4K dequantization formula. A Q4K weight is not a single value 
            each nibble nib4 (the raw 4-bit quantized weight, 0–15) must be multiplied by its 
            sub-block scale sc6u (6-bit integer, values 5–63) to recover the effective weight before dotting with the activation xi8. 
            The three operands are:
            xi8 (INT8) — the activation element from x
            nib4 (4-bit) — the raw weight nibble
            sc6u (6-bit) — the Q4K sub-block scale
            You cannot separate them because the effective weight nib4 × sc6u is never stored explicitly — 
            only the nibble and scale are stored in the GGUF format. The CPU does the same three-way 
            multiply during Q4K GEMV.

            "OPTION A"
    -------- 4→2 groups in mac_blocks_down_q4k
                The down Q4K path has 32 blocks per row. The current structure:

                for g in 0..3:          // 4 sequential groups
                    MAC_GROUP:
                        for n in 0..255:   // PIPELINE II=1 (256 cycles)
                            for bs in 0..7:  // UNROLL (8 blocks: g*8 + bs)
                                do MAC
                4 groups × 256 cycles = 1024 cycles of MAC per row, with 8 FP pipelines active simultaneously.

                Changing to 2 groups means:


                for g in 0..1:          // 2 sequential groups
                    MAC_GROUP:
                        for n in 0..255:   // PIPELINE II=1 (256 cycles)
                            for bs in 0..15:  // UNROLL (16 blocks: g*16 + bs)
                                do MAC
                2 groups × 256 cycles = 512 cycles per row — exactly half. But now 16 parallel MAC pipelines instead of 8. All arrays (rb, gate, sc6, mn6, int_acc_w/m) that used cyclic factor=8 must change to cyclic factor=16 or complete so the 16 simultaneously-unrolled instances don't fight for the same memory bank.

                Why didn't we do this from the start? With the old FP-based accumulators, 16 FP mul-add chains would have consumed ~800 DSPs and blown the budget. With the current fixed-point INT32 accumulators, each MAC chain is ~3 DSPs instead of ~20, so 16 chains ≈ 48 DSPs added — affordable.

    ---------Parallel loads inside compute_output
                Currently the DOWN_Q4K loop body executes this sequence for each pair of rows:


                load_row_down_q4k(W_down,  out_i,     row_buf0)   // 288 cycles — reads HP1
                load_row_down_q4k(W_down2, out_i + 1, row_buf1)   // 288 cycles — reads HP2
                mac_blocks_down_q4k(row_buf0, ...)                 // 1024 cycles
                mac_blocks_down_q4k(row_buf1, ...)                 // 1024 cycles
                Total per pair: 288 + 288 + 1024 + 1024 = 2624 cycles. Even though HP1 and HP2 are separate AXI ports, HLS schedules the two load calls sequentially because they're in a single function and share the loop body's control flow.

                What option B does: adds a DATAFLOW region inside compute_output with ping-pong (double) buffers so row N is being loaded while row N−1 is being MAC'd:


                Time →         [load row 0 from HP1] [load row 2 from HP1] [load row 4...]
                            [load row 1 from HP2] [load row 3 from HP2] [load row 5...]
                                                    [mac row 0]           [mac row 2]
                                                    [mac row 1]           [mac row 3]
                The two loads of a pair (HP1+HP2) happen simultaneously (different AXI ports). Then while those results are being MAC'd, the next pair loads. Per-pair cost collapses from 2624 to roughly max(288, 1024+1024) = 2048 cycles (the MACs are still sequential). With Option A also applied (MAC reduced to 512 each): max(288, 512+512) = 1024 cycles per pair — a 2.5× speedup over the current 2624.

                Why it's more complex: HLS DATAFLOW doesn't work inside a for-loop body directly. You'd need to restructure compute_output to use a #pragma HLS DATAFLOW region containing two sub-functions — a load_pair function and a mac_pair function — with ping-pong buffers (row_buf_A0/A1 for even iterations, row_buf_B0/B1 for odd). This is a known HLS pattern but requires careful buffer management and passing the correct buffer set to each call.

        The 6656-cycle extraction problem exists because GGUF stores Q6_K weights in interleaved block layout:

        [block 0: 128 ql bytes | 64 qh bytes | 16 sc bytes | 2 d bytes]  ← 210 bytes
        [block 1: 128 ql bytes | 64 qh bytes | 16 sc bytes | 2 d bytes]  ← 210 bytes
        ...
        [block 31: ...]
        So one row = 32 blocks × 210 bytes = 6720 bytes, all mixed together. The FPGA has to burst the whole row into flat[], then extract bytes one-by-one because block 0 ql[127] and block 1 ql[0] are not adjacent in memory.
