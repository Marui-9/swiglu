\begin{abstract}
  This paper presents a systems-driven approach to conversational audio
  inference on resource-constrained edge platforms by selectively offloading
  the heaviest computation phase of a liquid foundational generative
  conversational model to an FPGA.
  The integration is implemented in a modified llama.cpp/ggml-oriented flow
  through a fused operator, layer-aware weight caching, and interrupt-based
  host CPU-to-FPGA synchronization.
  The results shows that a mixed CPU+FPGA architecture improves
  performance-per-watt at low to moderate CPU thread counts, while higher
  thread counts remain limited by memory-transfer and orchestration overheads.
  These findings indicate that selective feed-forward acceleration is a
  practical and energy-effective approach for edge deployment of
  conversational generative workloads.

\keywords{Edge AI \and FPGA acceleration \and LFM2 \and hardware offload}
\end{abstract}

\section{Introduction, Research Question, and Contributions}\label{sec:intro}
Conversational audio generative AI workloads at the edge have to satisfy three
constraints at once: low latency, low power, and edge-constrained processing
ensuring privacy and reliability~\cite{reddi2025generative}.
Transformer-based generative backbones provide the required reasoning quality,
but their computational complexity remains expensive for embedded CPUs.
This issue has to be mitigated so that these systems can operate in real time
without relying on cloud assets.

We select the AMD-Xilinx Kria KV260 evaluation board to accelerate the AI
language backbone LFM2.5-1.2B-Thinking of the LFM2.5-Audio-1.5B model executed
by llama.cpp/ggml software.
This generative AI model was selected because it is edge-oriented.
Instead of mapping the full model to hardware, we profiled the CPU execution and
selective acceleration of the most complex block was adopted.
We address the following question: how conversational generative audio
workloads can be implemented to be energy-efficient on edge hardware while
achieving sufficient throughput under compute and memory-footprint
constraints.
The contributions are:
\begin{enumerate}
    \item Bottleneck-driven target selection. SwiGLU was identified as the main execution hotspot requiring hardware acceleration.
    \item CPU+FPGA integration. A fused hardware operator was implemented with graph-level interception, layer-aware weight caching, and interrupt-based completion.
    \item Efficiency/performance trade-off. CPU Only vs FPGA+CPU results are reported across 1 to 4 threads, showing performance-per-watt improvements and highlighting memory movement versus task orchestration.
\end{enumerate}

\section{Method}\label{sec:method}
\subsection{Profiling-Driven Offload Target}
We profiled the LFM2.5-1.2B-Thinking llama.cpp execution to identify where
most of the time is spent per token by the CPU. The MatMul + SwiGLU kernel
is dominant, accounting for about two-thirds of inference CPU execution time.
Therefore, feed-forward SwiGLU was mapped to the on-chip FPGA while all other
kernels remained on the CPU. This selection avoids a time-consuming full-model hardware design.

\subsection{CPU--FPGA Handoff in ggml}
Inference remains mostly CPU-based, and only the SwiGLU feed-forward step is
redirected to the FPGA. When execution reaches that step, the CPU prepares the
input data, starts the FPGA kernel, waits for completion, and then continues
the model with the returned output. Overhead was reduced by copying
feed-forward weights to shared memory once per layer and reusing them across
tokens. If input checks fail or the FPGA times out, the same step is
executed on the CPU, so inference proceeds with unchanged model behavior.

\subsection{Feed-Forward Accelerator Organization}
The FPGA accelerator implements:
$$ \text{FFN}(x) = (\text{SiLU}(x W) \odot (x V)) W_{down} $$
The internal pipeline follows the following structure: \textbf{Stage 1} loads
and replicates the INT8 input activation, \textbf{Stages 2a/2b} computes $X_1
= x W^{T}$ and $X_2 = x V^{T}$ in parallel, \textbf{Stage 3} applies gated
activation ($\mathrm{SiLU}(X_1)\cdot X_2$) with requantization, and
\textbf{Stage 4} performs the down projection with output write-back. Weights
are streamed from external memory; intermediate tensors are buffered on-chip
(BRAM/URAM) to sustain dataflow execution.

The compute-versus-bandwidth imbalance of the various stages was central to
the final system-level results and explain why kernel acceleration does not
translate into proportional workload speedup at higher CPU thread counts.

\section{Experiments and Results}\label{sec:results}
%\subsection{Experimental Setup}
Experiments were run on the Kria KV260 (quad-core ARM Cortex-A53 at 1.3\,GHz) with
embedded FPGA clocked at 250\,MHz featuring shared DDR memory. The software
stack was a modified \texttt{llama.cpp}/ggml runtime with optional fused
feed-forward offload. The LFM2.5-1.2B-Thinking backbone of the generative
audio model was tested in the Q4\_K format using \texttt{llama-bench}. Power
consumption was measured by the onboard telemetry (\texttt{xmutil}) during
each run.

The CPU+FPGA implementation improved both throughput and efficiency at low
thread counts (T1/T2), while the trend reversed at higher host parallelism
(T3/T4), where CPU Only performed better in tokens/s and tokens/s/W. Since
board-level power remains relatively close among modes, the dominant factor
is the effective throughput. This matches the method-level observation that
projection stages are memory-sensitive, while down-projection is more
compute-oriented.

\section{Conclusions}\label{sec:discussion_conclusion}
CPU+FPGA improves performance-per-watt at T1/T2, while CPU Only performs
better at T3/T4 because transfer and orchestration overheads dominate. This
matches the stage-level bottlenecks (the stage 2 operations are
bandwidth-limited; stage 4 is more compute-limited) and the FPGA resource
ceiling, where additional unrolling yields diminishing returns. Overall, the
approach was viable for energy-efficient edge inference. Next improvements are
expected to target overlap of transfer/compute, more regular memory access,
and higher weight/activation reuse.
