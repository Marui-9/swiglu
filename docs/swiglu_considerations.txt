## Practical architecture for KV260

A sequential phase structure works well:

`Phase 1: Load x (2048 F32) → BRAM (8 KB)
Phase 2: Stream W (quantized) → compute X1 = xW + b → URAM (32 KB)
Phase 3: Stream V (quantized) → compute X2 = xV + c → URAM (32 KB)
Phase 4: gate[j] = Swish(X1[j]) ⊙ X2[j], j=0..8191 (BRAM-only, ~8192 cycles)
Phase 5: Stream W_down (quantized) → compute output = gate · W_down → stream out`

**Buffer budget check (ZU5EV):**

- x vector: 8 KB → BRAM
- X1, X2 each: 32 KB → URAM (64 URAM288 blocks = 18.4 Mb available; 64 KB = ~2 URAMs)
- gate[]: 32 KB → reuse one of the URAM arrays

This fits comfortably. Phases 2/3/5 are DMA-sequential; Phase 4 is free-running and fast.

## Summary

| Aspect | Verdict |
| --- | --- |
| Three-stage decomposition | Correct |
| `#pragma HLS dataflow` across stages | Not useful here (single DMA bottleneck) |
| Dataflow within each matmul phase | Yes — same II=1 pipeline as linear_projection |
| LUT-based sigmoid | Correct |
| X1/X2 intermediate storage | URAM, ~64 KB total, feasible |
| True inter-stage parallelism | Only achievable with a second AXI DMA |

The cleanest path is a single HLS top-level function with five sequential phases, each internally pipelined to II=1. This reuses the pair-processing and 8-accumulator patterns from the working linear_projection kernel.

Before writing any HLS: confirm the weight quantization format (Q4_K vs Q6_K vs other) and the value of β — those two facts determine both the dequantization logic and the LUT design.

`general.architecture` decodes as `"lfm2"` (ASCII: 108=l, 102=f, 109=m, 50=2). β is not stored in GGUF metadata — it's hardcoded per architecture in llama.cpp. For the `lfm2` arch, and for all modern SwiGLU-based LLMs (Llama, Mistral, LFM2), β=1, making Swish = SiLU:

`SiLU(x) = x · σ(x)      where σ(x) = 1 / (1 + e^{-x})`

The LUT covers σ(x) only. Range of ±8 covers 99.97% of activation values at this scale.

## Quantization confirmed — and there's a mixed-format complication

| Tensor | Format | All 16 blocks? |
| --- | --- | --- |
| `ffn_gate.weight` (= W) | **Q4_K** | Yes, all 16 |
| `ffn_up.weight` (= V) | **Q4_K** | Yes, all 16 |
| `ffn_down.weight` (= W_down) | **Q6_K** | Only blocks 0, 1, 4, 7, 10, 13, 14, 15 |
| `ffn_down.weight` (= W_down) | **Q4_K** | Blocks 2, 3, 5, 6, 8, 9, 11, 12 |

This is the standard Q4_K_M pattern: important/high-sensitivity layers keep Q6_K for `ffn_down`. The hardware IP must handle **both formats for ffn_down**.

## Q4_K block format (for reference)

`block_q4_K (144 bytes total):
  [0..1]    d     : fp16 super-block scale for scale factors
  [2..3]    dmin  : fp16 super-block scale for min values
  [4..15]   scales: 12 bytes — 8 scale values + 8 min values, 6 bits each
  [16..143] qs    : 128 bytes — 256 weights as 4-bit nibbles, 2 per byte

Dequant formula for weight n in block:
  nibble = qs[n/2] >> ((n%2)*4)) & 0xF         // 4-bit unsigned
  scale  = scales[n/32] (decoded from 6-bit)
  min    = mins[n/32]   (decoded from 6-bit)
  w      = d * scale * nibble - dmin * min`

Compare to Q6_K (210 bytes): the key difference is Q4_K uses a `min` subtraction (affine, not zero-centered), so each weight needs both a scale **and** a min value. The 6-bit scale packing in the 12-byte field is the tricky part.

## Implementation consequences

1. **Phases 2 & 3** (ffn_gate, ffn_up): pure Q4_K, same design for both, 144 bytes/block
2. **Phase 5** (ffn_down): needs a mode register in the AXI-Lite control port — software writes Q4K or Q6K mode before starting; hardware switches the dequant path. This is simpler than two separate IPs.
3. **Block sizes for pair-processing:** Q4_K pairs: 2×144 = 288 = 72 words exactly (integer — pairs work cleanly). Q6_K pairs: 2×210 = 420 = 105 words (same as linear_projection — already proven).
4. **Architecture metadata** from beta.txt also confirms: `lfm2.feed_forward_length = 8192`, `lfm2.embedding_length = 2048`, `lfm2.block_count = 16` — matches the expected dimensions.
