 
FILES AND EXECUTION PROCESS OF LLAMA.cpp

 ggml.h -> no execution logic, tensor structs, enums for ops, builder prototypes, declarations
 ggml.c -> contains graph building helpers: it constructs the compute graph and returns
            tensors representing nodes.
 ggml-cpu.c -> executor of cpu backend: threads setup, graph planning, ggml_compute_forward 
            switch that dispatches the per-op kernels. Here are the functions called by the graph runner.
            the offload logic (UIO mmap, reg programming, IRQ wait, copy out) must live here in a kernel.

End-to-end flow (CPU backend):

1. Model is loaded from GGUF; llama.cpp (or the app) calls ggml factory functions (in ggml.c) to 
    build a compute graph of tensors with op and inputs wired.
2. ggml_graph_compute (in ggml-cpu.c) plans the graph, spins a threadpool.
3. Each node is visited in topological order; for each node, ggml_compute_forward switch picks the kernel.
4. The selected kernel executes (CPU math or your FPGA offload) and writes results to the node’s tensor 
    data. Threads sync at a barrier before the next node.

Note that ggml-cpu.c has a massive switch case statement at line 1686:
its purpose is to build a GGML compute graph of the model, so that 
execution can jump between estabilished kernels (functions)
[ggml-cpu.c:1698–2058]  see swiglu_algorythm.txt for block specifics

Each worker runs ggml_graph_compute_thread (ggml-cpu.c:2952–3007), 
itegrating graph nodes in topological order. After each node, all threads hit 
ggml_barrier so dependencies are respected.
    For the current node, ggml_compute_forward dispatches to the 
    op-specific kernel (ggml-cpu.c:1698–2058).
Per‑op parallelism: before execution, ggml chooses n_tasks for that op 
    (ggml-cpu.c:2200–2268); many heavy ops (e.g., MUL_MAT, GLU/SwiGLU) use n_threads.
    Inside each kernel, loops are partitioned by ith/nth from ggml_compute_params. 
    Example: ggml_compute_forward_mul_mat tiles rows/cols and each thread processes 
    a slice based on ith (ggml-cpu.c:1235+). So multiple threads run the same node 
    concurrently on disjoint chunks, then synchronize at the barrier before moving 
    to the next node.

 ggml_graph_compute [ggml-cpu.c:3238–3294] builds the sequence of functions 
 to be executed, starts threads and then each thread calls 
 ggml_compute_forward, which makes the thread do whatever operation is needed.

"batch" (or "second dimension") = number of tokens processed together 
                                    (at the same time)
in ggml the second dimension (ne[1]) is 1 for decode=generation
and >1 (many tokens in parallel) for prefill=prompt. For the LFM2 MLP, x 
is shaped [2048, n_tokens].

To truly overlap FPGA with other work you’d need a larger scheduler change 
(asynchronous node completion / dependency tracking), which ggml doesn’t provide today. 
A lighter alternative is to overlap inside the kernel (double-buffer x/out and let the 
same thread enqueue next chunk while FPGA processes previous), but graph-level parallel 
overlap isn’t available without deeper changes.

Where it can fail in the current logic:

Graph construction: llama.cpp builds the FFN as separate W/V/silu/mul/W_down ops. 
There’s no call to ggml_swiglu_fused_hw, and the GGUF loader doesn’t rewrite nodes to the fused op. 
Result: no GGML_OP_SWIGLU_FUSED_HW tensors in the graph.
Fallback guard inside fused kernel: even if a node existed, the fused kernel early‑returns for any 
shape/type mismatch (x must be F32, gate/up Q4_K, down Q4_K/Q6_K, exact 2048/8192 dims). 
The standard Q4_K_M and Q6_K models satisfy this, so the main blocker is the missing node.
